# 小红书线索同步 - 多维表格 - 边栏插件

---

## 产品背景

使用小红书进行获客的商家，通常会在私信中收到大量线索（如留资、咨询、意向客户等）。以往需要人工从小红书后台或私信里复制信息，再粘贴到 Excel 或协作表格中整理，既耗时又容易遗漏或出错。

**小红书线索同步** 面向的正是这类场景：已开通小红书「私信 API 对接」、并希望用**飞书多维表格**统一管理线索的团队或个人。通过本插件，可将小红书推送的私信线索自动写入当前多维表格中的「小红书线索」表，无需再手动复制粘贴；配合飞书的视图、权限和自动化，可进一步实现线索自动分配、提醒与跟进，提升获客到跟进的效率。

---

## 产品功能

- **一键建表**：在当前多维表格下创建固定表「小红书线索」，包含与小红书推送一致的 38 个字段（如 id、time、nick_name、phone_num、wechat、来源与投放信息、客服与渠道等）；若表已存在但缺字段，支持一键补全，无需删表重建。
- **授权与配置**：在插件内提交多维表格授权码后，由服务器保存该 Base 与表的对应关系；每个多维表格（baseId）独立配置，多 Base 可共用同一台服务器，线索按 Base 写入各自表。
- **Webhook 接收与写入**：插件会生成该 Base 专属的 Webhook 地址；将该地址填到小红书「私信 API 对接」的消息接收地址后，小红书在有新线索（有效私信标注与广告归因）时会 POST 推送，服务器接收后自动写入对应多维表格，实现**实时同步**。
- **形态说明**：产品由**多维表格边栏插件**（前端，负责建表、检测结构、提交授权码、展示 Webhook）和**服务器**（后端，接收配置与小红书推送、调用 Base API 写表）组成；授权码仅存于服务器侧，保证数据与权限安全。

## 使用说明

### 使用前准备

- **小红书侧**：已开通「私信 API 对接」，并可在小红书聚光平台配置「消息接收地址」。
- **飞书侧**：使用支持多维表格边栏插件的环境；当前用户对该多维表格有编辑权限（可建表、写表）。
- **网络**：部署的服务器需可被小红书访问（建议 HTTPS）；若插件页面为 HTTPS，服务器地址也须为 HTTPS。

### 添加插件到多维表格

- 新建或打开任意多维表格，点击 **「插件」** 展开插件面板。
- 点击 **「自定义插件」** → **「+ 新增插件」**，在输入框中填入本插件的运行地址，点击 **「确定」** 完成添加并运行插件。

### 创建线索底表

- 打开插件后，若当前多维表格中**没有**「小红书线索」表：点击 **「创建同步表」**，插件会在当前 Base 下创建「小红书线索」表并添加全部 38 个字段。
- 若**已有**该表但缺少部分字段：插件会提示「表已存在，但结构不完整」并列出缺失字段，点击 **「补全缺失字段」** 即可自动补全，无需删表。

### 提交插件授权码

- 在多维表格中进入 **「插件 → 自定义插件 → 获取授权码」**，复制授权码。
- 在本插件内粘贴授权码，点击 **「提交」**。提交成功后，服务器会保存当前 Base 与表的配置，该 Base 的线索将写入「小红书线索」表。

### 复制 Webhook 地址到小红书聚光平台

- 授权码提交成功后，插件会显示 **Webhook 地址**（形如：`https://<服务器域名>/webhook/leads/<baseId>`）。
- 复制该地址，在小红书**聚光平台**的 **「私信 API 对接」** 中，将其填写为 **消息接收地址** 并保存。
- 保存后，新线索会由小红书自动推送到该地址，服务器将数据写入「小红书线索」表。

完成以上四步后，新线索会实时出现在多维表格中，无需再手动整理私信。

> **提示：关于线索重复**  
> 由于小红书数据推送规律，一条线索会分 2 次推送（第一次包含线索基础信息，第二次为广告数据回传）。若表格中出现同一线索多条记录，为正常现象，建议在业务表或视图中按线索 ID 等字段进行去重后再使用。

### 常见问题

| 问题 | 说明 |
|------|------|
| 提示表名已存在 | 插件会检测并提示补全缺失字段；若需重新建表，请先删除或重命名现有「小红书线索」表。 |
| 提交授权码失败 | 请检查网络、服务器地址是否可访问；若部署在内网，需确保当前环境能访问该服务器。 |
| 小红书已推送但表格无数据 | 确认 Webhook 地址与当前 Base 的 baseId 一致、步骤 2 已提交成功；可查看服务器日志排查 4xx/5xx 或 Base API 报错。 |
| 如何获取 app_token / table_id | 可从多维表格 URL 中查看，或使用官方「开发工具」等插件获取；本插件以 baseId 作为租户标识，Webhook 路径中已包含 baseId。 |
| 为什么同一条线索有多条记录 | 小红书会分 2 次推送（线索基础信息 + 广告归因），属于正常现象，建议在业务表或视图中按线索 ID 去重后使用。 |

## 插件上架到多维表格的步骤

### 1. 本地打包与产物配置

- **部署方式**：平台会直接使用你提供的**静态产物**进行部署（也可使用自有服务器，提供运行地址）。需在 `package.json` 中指定构建产物目录，例如：
  ```json
  {
    "output": "dist"
  }
  ```
  表示上传并部署 `dist` 目录。

- **产物纳入仓库**：从 `.gitignore` 中**去掉 `dist`**，确保提交的仓库内包含构建后的静态文件。每次代码更新后执行 `npm run build`，将新的 `dist` 一并提交。

- **资源路径**：打包产物的资源引用**必须使用相对路径**，不能使用绝对路径。若使用 Vite，在 `vite.config.js` 中设置：
  ```js
  import { defineConfig } from "vite";
  export default defineConfig({
    base: "./",  // 使用相对路径
    // ...
  });
  ```

- **提交前检查**：仓库中需至少包含：
  - 静态产物目录（通常为 `dist`）；
  - `package.json` 中存在 `output` 且值为该目录（如 `"output": "dist"`）。

### 2. 路由模式

- **禁止使用 history 路由**，请使用 **hash 路由**（如 `vue-router` 的 `createWebHashHistory()` 或 React Router 的 `HashRouter`），避免平台部署后路径 404。

### 3. 填写共享表单

- 按多维表格插件上架流程，在平台填写并提交共享/上架表单，提供仓库地址、运行说明等所需信息。

### 4. 审核通过后的部署与运行要求

- 审核通过后，平台会 **Fork 并部署**你的项目与配置。
- **运行时要求**：插件需能基于当前多维表格的数据结构进行初始化配置；若无法获取正确数据结构，或**必须依赖用户选中单元格**才能工作，则需在界面给出**明确提示**，引导用户如何操作（例如「请先打开某张表」或「请先选中某一单元格」），避免用户产生疑惑。

## 小红书私信线索字段说明

依据 [小红书私信数据推送 API 对接文档](https://fe-video-qc.xhscdn.com/fe-platform-file/104101b831o4jt31h06s68p465hi00000000002njnvv4o.pdf) 整理，POST 请求体为 JSON，根级含 `data`、`timestamp`、`source`。

### 根级字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `data` | object | 线索详情，见下表 |
| `timestamp` | number | 时间戳（毫秒） |
| `source` | string | 固定为 `"小红书"` |

| `time` | string | 私信线索创建时间 | `"2023-08-28 13:00:00"` |
| `red_id` | string | 小红书 ID | `"100563573"` |
